### Mock pattern established by the project

From `tests/accounts-actions.test.ts` — the project's canonical pattern for mocking the Supabase client in server action tests:

```typescript
vi.mock('@/lib/supabase/server', () => ({
  createClient: vi.fn(),
  getUser: vi.fn(),
}))

vi.mock('next/cache', () => ({
  revalidatePath: vi.fn(),
}))

// Import AFTER mock declarations
import { createClient, getUser } from '@/lib/supabase/server'
```

The Supabase client is replaced with a fake object whose `.from(table)` returns a chainable proxy that resolves with canned `{ data, count, error }` payloads. The proxy's methods (`select`, `insert`, `eq`, `in`, `or`, `ilike`, `order`, `range`, `maybeSingle`, etc.) all return `this` (the same proxy) to enable chaining. The proxy becomes a thenable by attaching `.then` directly.

### Server action signatures and return types (from calls/actions.ts)

```typescript
// Filters
export interface UnifiedCallFilters {
  page?: number
  pageSize?: number
  type?: 'all' | 'ai' | 'human'
  direction?: 'all' | 'inbound' | 'outbound'
  missed?: boolean
  q?: string
  from?: string
  to?: string
}

// Return types
export interface UnifiedCallsResult {
  rows: UnifiedCallWithContact[]
  total: number
  page: number
  pageSize: number
}

export async function getUnifiedCalls(
  filters: UnifiedCallFilters = {},
): Promise<UnifiedCallsResult>

export async function getUnifiedCall(id: string): Promise<UnifiedCallWithContact | null>
```

### Key implementation details that tests must reflect

`getUnifiedCalls` internal flow:
1. Calls `getUser()` — returns early `{ rows: [], total: 0, page, pageSize }` if null
2. Calls `createClient()` to get Supabase client
3. Queries `.from('unified_calls').select('*', { count: 'exact' }).order(...).range(...)`
4. Applies eq/in/or filters conditionally based on `filters.*`
5. On error or empty data: returns `{ rows: [], total: 0, page, pageSize }`
6. Collects unique `contact_id` values from rows
7. If any contact_ids present: queries `.from('contacts').select('id, name, phone, email').in('id', contactIds)`
8. Returns `{ rows: UnifiedCallWithContact[], total: count, page, pageSize }`

`getUnifiedCall` internal flow:
1. Calls `getUser()` — returns null if no user
2. Queries `.from('unified_calls').select('*').eq('id', id).maybeSingle()`
3. Returns null if `data` is null (not found)
4. If `data.contact_id` is truthy: queries `.from('contacts').select('id, name, phone, email').eq('id', ...).maybeSingle()`
5. Returns `{ ...data, contact }` where contact is the contact row or null

### How to build the fake Supabase client for these tests

The fake client needs to handle two tables: `unified_calls` and `contacts`. Build a helper function `buildFakeClient(responses)` that accepts per-table canned responses and returns an object with a `from(table)` method. Each table's proxy supports the chain methods used by the actions:

For `unified_calls`: `select`, `order`, `eq`, `in`, `or`, `range` — resolves with `{ data, count, error }`
For `contacts`: `select`, `in`, `eq`, `maybeSingle` — resolves with `{ data, error }`

The proxy pattern from `accounts-actions.test.ts` (chainable thenable) works directly. Copy and adapt it.

**Implementation guidance for `buildFakeClient`:**

The fake client must handle the call chains used internally by the actions. The key challenge is that `getUnifiedCalls` chains: `.from('unified_calls').select('*', {count:'exact'}).order(...).eq(...).in(...).or(...).range(...)` and then awaits. The `getUnifiedCall` chains: `.from('unified_calls').select('*').eq('id', id).maybeSingle()` and awaits.

Use the chainable proxy pattern: every method returns `this`, and `then` resolves with the canned result. This works because `await proxy` calls `proxy.then(resolve, reject)`.

For `contacts` in `getUnifiedCalls`, the second query is: `.from('contacts').select(...).in('id', contactIds)`. In `getUnifiedCall`, the contact query is: `.from('contacts').select(...).eq('id', contact_id).maybeSingle()`.

The simplest approach: build two separate proxies (one for `unified_calls`, one for `contacts`) and have `from(table)` return the correct proxy based on the table name.

Since `getUnifiedCall` calls `.maybeSingle()` instead of awaiting the chain directly, make `.maybeSingle()` return a Promise resolving to the canned result (not the proxy itself).
