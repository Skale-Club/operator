# Plan 93-02 — Redis rate limiter for /book/*

## Objective

Add per-IP rate limiting on the public booking flow (`createBooking`) using the existing Upstash Redis client. Fail-open if Redis is unreachable.

## Tasks

### Task 1 — Rate limiter helper (auto)

Create `src/lib/rate-limit.ts`:

- Export `rateLimit(key: string, limit: number, windowSeconds: number): Promise<{ allowed: boolean; remaining: number; resetAt: number }>`
- Use `INCR` + first-call `EXPIRE` (fixed-window counter; self-cleaning)
- If `redis.isReady === false`, log a warning and return `{ allowed: true, remaining: limit, resetAt: 0 }` (fail-open)
- Wrap all Redis calls in try/catch; on error log + fail-open

### Task 2 — Wire into createBooking (auto)

In `src/app/(dashboard)/scheduling/_actions/bookings.ts`:

- Read the client IP inside `createBooking` via `headers()` from `next/headers` (server action context) — try `x-forwarded-for`, then `x-real-ip`, fall back to `'unknown'`
- Call `rateLimit('booking:' + ip + ':' + event_type_id, 5, 3600)` BEFORE the slot conflict pre-check
- If `!allowed`, return `{ ok: false, error: 'rate_limited' }`
- Update `BookingForm` to surface a friendly toast on `rate_limited`

## Verification

- `src/lib/rate-limit.ts` exists and exports `rateLimit`
- `createBooking` invokes `rateLimit` and returns `rate_limited` when exceeded
- `BookingForm` handles `rate_limited` distinctly from `slot_taken`
- `npm run build` exits 0

## Done

- Booking page is rate-limited at 5/hour per (IP, event_type_id)
- Booking still works when Redis is offline
