# Incident: Dashboard render error digest `1621801304`

## Symptoms

- Home dashboard (`/`) renders the Next.js production error page.
- Browser stack trace shows `i4 ‚Üí us ‚Üí i4 ‚Üí us` repeating thousands of times ‚Äî a stack overflow from infinite recursion, NOT a normal async throw.
- Previous fix attempts that wrapped data fetches in `try/catch` did NOT help, confirming the issue is **structural** (render tree), not data-related.

## Stabilization commit

Replaced the home dashboard with a minimal version and made the dashboard error boundary dependency-free.

Files changed:
- `src/app/(dashboard)/error.tsx` ‚Äî stripped to pure HTML + inline styles. No imports from the dashboard tree. This breaks any feedback loop where the error boundary itself depends on the broken provider/token chain.
- `src/app/(dashboard)/page.tsx` ‚Äî replaced rich dashboard (metrics, activity feed, pipeline widget, welcome checklist) with a minimal greeting + 5 quick links.

## Suspect ranking (to investigate after production is stable)

### High likelihood

1. **`<link rel="icon" href={branding.logoUrl} />` inside dashboard layout body** (line 92-95 of original `layout.tsx`). This is a bare `<link>` outside `<head>`; Next.js Float will hoist it. If `branding.logoUrl` ever returns something that triggers re-fetch on every render (e.g. a signed URL with a per-request timestamp), it could re-hoist and force a layout re-render. **Check whether `getOrgBranding` is memoised across the request and whether `logoUrl` is stable.**

2. **`Suspense` boundaries around server components that throw synchronously in the SAME render frame as their fallback.** The home page has 4 nested `<Suspense>` boundaries (HomeMetrics, ActivityChartSection, ActivityPanel, PipelineWidget). If any of these server components throws BEFORE awaiting (e.g. an import-time module error, or a synchronous `createClient()` throw), the Suspense boundary might trigger the segment error boundary, which then re-renders the page, which throws again.

### Medium likelihood

3. **`OnboardingTour` mounted in layout** (not on the failing page directly, but it's mounted everywhere). `useEffect(setTimeout(setActive, 600))` runs on every pathname change. Safe in isolation, but combined with `PageTransition` (also keyed on pathname) it could create a route-transition feedback loop on very fast re-renders.

4. **`PageTransition` with `AnimatePresence mode="wait"`** ‚Äî if `pathname` stays the same but the child's identity changes (e.g. a Suspense fallback flickering in/out), AnimatePresence could re-mount children in a loop. Known framer-motion edge case.

### Lower likelihood (already audited, no obvious issue)

5. CelebrationProvider ‚Äî value memoised.
6. TwilioDeviceProvider ‚Äî gated behind `enabled` prop from the server; the prop is stable.
7. BrandingStyle ‚Äî pure.

## Why digest `1621801304` specifically points to a recursion loop

In React production builds, function names are mangled to `i4`, `us`, etc. A normal React error stack shows ~5‚Äì10 frames before the `renderRootSync` boundary. THOUSANDS of `i4 ‚Üí us ‚Üí i4 ‚Üí us` repeating frames means the same two functions called each other (or were called inside each other's reconciliation cycle) until the JS engine ran out of stack ‚Äî a hallmark of:

- Error boundary catching ‚Üí render ‚Üí throw ‚Üí catch ‚Üí render ‚Üí throw
- A component setting state inside its own render (no useEffect wrapper)
- A `use(promise)` call where the promise resolves into another `use(promise)`

The fact that wrapping data fetches in try/catch did NOT fix it rules out option 3 (any data error would have been caught). That leaves option 1 (the error.tsx + page.tsx loop) and option 2 (a setState in render) as the most likely culprits ‚Äî both of which the stabilization commit defuses, because `error.tsx` is now provider-free and `page.tsx` is minimal.
