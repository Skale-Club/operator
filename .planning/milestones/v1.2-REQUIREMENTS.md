## v1.2 Requirements

### BRAND — Platform Rename

- [x] **BRAND-01**: Platform UI displays "Operator" name instead of "VoiceOps" across all pages, navigation, and branding elements — **Complete** (Phase 1)
- [x] **BRAND-02**: Page titles, sidebar, login page, and any hardcoded references updated to Operator — **Complete** (Phase 1)

### WIDGET — Embeddable Chat Widget

- [x] **WIDGET-01**: Admin can install the chat widget on any third-party site using a single `<script>` tag (no framework dependency on host) — **Complete** (Phase 4)
- [x] **WIDGET-02**: Script tag is GTM-compatible (loads asynchronously, no blocking) — **Complete** (Phase 4)
- [x] **WIDGET-03**: Widget renders as a floating chat bubble that expands into a full chat panel — **Complete** (Phase 4)
- [x] **WIDGET-04**: Widget is identified per-org via a public token embedded in the script tag — **Complete** (Phase 4)
- [x] **WIDGET-05**: Widget works without visitor login or authentication — **Complete** (Phase 4)

### CHAT — Conversation Engine

- [x] **CHAT-01**: Visitor can send messages and receive streamed AI responses in real time (SSE) — **Complete** (Phase 3)
- [x] **CHAT-02**: AI responses draw from the org's knowledge base (LangChain SupabaseVectorStore) — **Complete** (Phase 3)
- [x] **CHAT-03**: AI can call org tools during conversation (action engine `executeAction`) — **Complete** (Phase 3)
- [x] **CHAT-04**: Conversation context is maintained within a session using Redis short-term memory — **Complete** (Phase 2)
- [x] **CHAT-05**: Conversation history is persisted to Supabase long-term memory (per org, per session) — **Complete** (Phase 2)
- [x] **CHAT-06**: Each conversation session is identified by a unique session ID (anonymous visitor) — **Complete** (Phase 2)

### ADMIN — Widget Configuration

- [x] **ADMIN-01**: Admin can configure widget appearance per org (display name, primary color, welcome message) — **Complete** (Phase 5)
- [x] **ADMIN-02**: Admin page shows the embed `<script>` tag ready to copy — **Complete** (Phase 5)
- [x] **ADMIN-03**: Admin page shows a live preview of the widget with current configuration — **Complete** (Phase 5)
- [x] **ADMIN-04**: Admin can regenerate the org's widget public token (invalidates old installs) — **Complete** (Phase 5)

### INFRA — Backend Infrastructure

- [x] **INFRA-01**: Redis connection configured and available for chat session storage — **Complete** (Phase 1)
- [x] **INFRA-02**: Supabase schema includes `chat_sessions` and `chat_messages` tables with RLS (later renamed to `conversations`/`conversation_messages` in migration 015) — **Complete** (Phase 1 + Phase 6)
- [x] **INFRA-03**: Public-facing chat API route (`/api/chat/[token]`) validates org token and scopes all queries to the org — **Complete** (Phase 2)
- [x] **INFRA-04**: Widget asset served from the platform's own domain (no external CDN dependency required) — **Complete** (Phase 1)

### INBOX — Chat Admin Inbox (Phase 6, scope expansion)

> **Note:** INBOX requirements were added as Phase 6 expanded scope post-initial-plan. They were delivered and verified but not formally added to the original REQUIREMENTS.md.

- [x] **INBOX-01**: Admin can view a list of all widget conversations for their org — **Complete** (Phase 6)
- [x] **INBOX-02**: Conversation list shows visitor info, last message preview, and status — **Complete** (Phase 6)
- [x] **INBOX-03**: Admin can filter conversations by status (open/closed) — **Complete** (Phase 6)
- [x] **INBOX-04**: Admin can search conversations — **Complete** (Phase 6)
- [x] **INBOX-05**: Admin can view full message thread for a conversation — **Complete** (Phase 6)
- [x] **INBOX-06**: Admin can reply to a conversation from the inbox — **Complete** (Phase 6)
- [x] **INBOX-07**: Widget settings relocated under Chat section in sidebar — **Complete** (Phase 6)
